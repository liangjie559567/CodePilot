## 关键冲突与仲裁

### 冲突 1: v1.0 是否应包含协作功能？

**Tech Lead**: 砍掉协作功能，技术复杂度降低 35%，18 周可行  
**Product Director**: 无协作功能缺少差异化，建议添加轻量协作（代码片段分享）  
**UX Director**: 支持 Product Director，但需要 2 周额外设计时间

**仲裁结果**: ✅ **采纳 Product Director 建议**
- v1.0 添加轻量协作功能（代码片段分享 + 评论）
- 砍掉实时协作、权限管理、冲突解决
- 总周期延长至 20 周（18 周 + 2 周轻量协作）

**理由**:
- 纯性能优化无法形成市场差异化
- 轻量协作开发成本可控（2 周）
- 产品定位更清晰："高性能 + 隐私优先的协作工具"

---

### 冲突 2: Phase 0 Week 5-6 的设计范围

**UX Director**: 不应设计"协作和垂直领域功能"（已砍掉），应聚焦 v1.0 实际功能  
**Tech Lead**: 同意，Week 5-6 应聚焦安全加固和 POC 验证  
**Product Director**: 如果添加轻量协作，Week 5-6 需要设计代码片段分享 UI

**仲裁结果**: ✅ **采纳 UX Director 建议 + Product Director 补充**
- Week 5-6 删除"协作和垂直领域功能设计"
- 聚焦 v1.0 实际功能：加载状态、错误提示、性能感知、轻量协作 UI
- 补充详细设计交付物清单

---

### 冲突 3: AI 模型架构决策

**Tech Lead**: 必须在 Week 7 前明确（本地 vs 云端）  
**Critic**: 云端 API 存在数据泄露风险，建议本地模型  
**Product Director**: 本地模型增加 4-6 周开发时间，建议 v1.0 使用 Claude API

**仲裁结果**: ✅ **采纳 Product Director 建议**
- v1.0 使用 Claude API（云端）
- 砍掉 ONNX 量化方案（节省 3 周）
- v1.1 添加本地模型支持（隐私敏感用户）

**理由**:
- 本地模型开发成本高（4-6 周）
- Claude API 性能和质量更稳定
- 80% 用户可接受云端 API

---
### 冲突 4: 实施周期评估

**Tech Lead**: 18 周基本现实，建议延长至 19 周  
**Product Director**: 建议 20 周（添加轻量协作）  
**UX Director**: 支持 20 周（需要 2 周设计轻量协作）

**仲裁结果**: ✅ **采纳 Product Director 建议 - 20 周**
- Phase 0: 3 周（安全加固 + POC + 轻量协作设计）
- Phase 1: 4 周（Tree-sitter + 向量检索）
- Phase 2: 3 周（Claude API 集成 + p-queue）
- Phase 2.5: 2 周（轻量协作功能）← 新增
- Phase 3: 4 周（集成测试 + 优化 + 安全审计）
- Phase 4: 4 周（Beta 测试 + 迭代 + 发布准备）

## 修订后的最终实施计划

### Phase 0: 安全加固与设计（Week 1-3）

**Week 1-2: 安全加固**
- 修复 5 个 P0 安全漏洞
