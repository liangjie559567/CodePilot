## 1. Security Audit (安全审计)

### 1.1 P0 漏洞修复评估 ✅ Pass

原评审发现的 5 个 P0 漏洞均已纳入 Week 5-6 修复计划：

| 漏洞 | 原风险等级 | 修复方案 | 评估 |
|------|-----------|---------|------|
| Tree-sitter 文件大小无限制 | P0 | 添加 10MB 上限 | ✅ 充分 |
| AST 深度无限制 | P0 | 添加 500 层上限 | ✅ 充分 |
| 路径遍历攻击 | P0 | 路径白名单验证 | ✅ 充分 |
| 解析队列无限流 | P0 | 最多 5 个并发解析 | ⚠️ 需要动态调整 |
| 向量维度攻击 | P0 | 维度验证 | ⚠️ 验证逻辑不足 |

**评估细节**:

1. **文件大小限制 (10MB)** - ✅ 充分
   - 10MB 对单文件代码已足够（Linux kernel 最大单文件 ~8MB）
   - 建议添加配置项允许高级用户调整

2. **AST 深度限制 (500 层)** - ✅ 充分
   - 正常代码嵌套深度 < 20 层，500 层足够防御恶意构造
   - 建议添加深度超限时的友好错误提示

3. **路径白名单验证** - ✅ 充分
   - 防止 `../../etc/passwd` 类攻击
   - 建议使用 `path.resolve()` + `path.relative()` 双重验证

4. **解析队列限流 (5 个并发)** - ⚠️ 需要动态调整
   - 固定 5 个并发在低配设备上仍可能导致卡顿
   - **建议**: 根据 CPU 核心数动态调整 (Math.min(cpuCount, 5))
   - **建议**: 添加内存压力检测，超过 80% 时暂停解析

5. **向量维度验证** - ⚠️ 验证逻辑不足
   - 仅验证维度不足以防御注入攻击
   - **缺失**: 向量范数验证 (L2 norm 应在合理范围内)
   - **缺失**: 异常值检测 (检测全 0 或全 1 向量)
   - **建议**: 添加向量合法性检查函数

```typescript
function validateVector(vec: number[], expectedDim: number): boolean {
  if (vec.length !== expectedDim) return false;
  const norm = Math.sqrt(vec.reduce((sum, v) => sum + v * v, 0));
  if (norm < 0.1 || norm > 10) return false; // 范数异常
  const zeros = vec.filter(v => v === 0).length;
  if (zeros > expectedDim * 0.9) return false; // 过多零值
  return true;
}
```

### 1.3 新增安全风险 ⚠️ Conditional Pass

砍掉协作功能后，出现新的安全关注点：

#### 1.3.1 模型架构不明确导致的数据泄露风险 - P1

**问题**: 综合评审报告中"模型架构未明确（本地 vs 云端）"

**风险分析**:
- 如果使用 Claude API：用户代码会发送到 Anthropic 服务器
- 如果用户处理敏感代码（金融、医疗、军工），存在合规风险
- GDPR/CCPA 要求明确告知数据处理方式

**修复建议**:
1. **Week 5 必须明确模型架构**
2. 如果使用云端 API：
   - 添加隐私政策弹窗（首次启动时）
   - 提供"敏感模式"（禁用云端推理）
   - 添加代码脱敏选项（仅发送函数签名，不发送实现）
3. 如果使用本地模型：
   - 添加模型签名验证（防止模型投毒）
   - 添加模型文件完整性检查

#### 1.3.2 向量索引持久化的安全风险 - P2

**问题**: FAISS 索引存储在本地磁盘，可能被恶意篡改

**攻击场景**:
1. 攻击者替换 `.faiss` 索引文件
2. 注入恶意向量，指向钓鱼网站或恶意代码
3. 用户搜索时被引导到恶意内容

**修复建议**:
- 添加索引文件 HMAC 签名
- 启动时验证索引完整性
- 检测到篡改时重建索引

```typescript
import crypto from 'crypto';

function signIndex(indexPath: string, secretKey: string): string {
  const data = fs.readFileSync(indexPath);
  return crypto.createHmac('sha256', secretKey).update(data).digest('hex');
}

function verifyIndex(indexPath: string, signature: string, secretKey: string): boolean {
  const expected = signIndex(indexPath, secretKey);
  return crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected));
}
```

## 2. Edge Case Analysis (边缘场景分析)

### 2.1 大规模项目边缘情况 ⚠️ Conditional Pass

#### 2.1.1 超大型项目 (100K+ 文件) - ⚠️ 部分覆盖

**场景**: Chromium (300K 文件), Android AOSP (500K 文件)

**现有方案**:
- Week 7 POC 验证包含 10K/50K/100K 文件测试
- 分片构建和渐进式加载

**缺失风险**:
1. **索引构建时间过长** - 100K 文件可能需要数小时
   - **影响**: 用户首次打开项目时长时间无响应
   - **建议**: 添加后台构建 + 部分可用策略
   - **建议**: 显示预计剩余时间和取消按钮

2. **磁盘空间耗尽** - FAISS 索引可能占用数 GB
   - **影响**: 低配设备磁盘空间不足导致崩溃
   - **建议**: 构建前检查可用空间 (至少 5GB)
   - **建议**: 提供"轻量模式"（仅索引常用文件）

3. **内存溢出** - 修订目标 600-800MB 仍可能不足
   - **影响**: 在 4GB 内存设备上崩溃
   - **建议**: 添加内存压力监控，超过阈值时卸载缓存

#### 2.1.2 深层嵌套目录 (>20 层) - ✅ 已覆盖

**场景**: node_modules 深层依赖，Java 包路径

**现有方案**: 路径白名单验证

**评估**: ✅ 充分，建议添加路径长度限制 (Windows 260 字符)

---

### 2.2 低配设备边缘情况 ⚠️ 覆盖不足

#### 2.2.1 低内存设备 (4GB RAM) - ⚠️ 高风险

**场景**: 用户在 4GB 内存的笔记本上打开中型项目 (10K 文件)

**风险分析**:
- 修订目标内存占用 600-800MB
- 加上 Electron (200MB) + 操作系统 (2GB) + 浏览器 (1GB)
- 剩余可用内存 < 1GB，可能触发 OOM

**缺失防御**:
1. **启动时内存检测** - 检测系统可用内存，低于 2GB 时警告
2. **动态降级策略** - 内存不足时自动禁用向量检索
3. **优雅降级而非崩溃** - OOM 时保存状态并提示重启

#### 2.2.2 低性能 CPU (双核) - ⚠️ 中等风险

**场景**: 用户在双核 CPU 设备上使用

**风险分析**: 固定 5 个并发解析会占满 CPU，导致 UI 卡顿

**修复建议**: 动态调整并发数 = Math.min(cpuCount, 5)

---

### 2.3 用户误操作边缘情况 ✅ Pass

#### 2.3.1 重复点击/快速切换 - ✅ 已覆盖

**现有方案**: 解析队列限流 (5 个并发)

**评估**: ✅ 充分，建议添加防抖 (debounce 300ms)

#### 2.3.2 网络中断 (云端 API) - ⚠️ 需要明确

**场景**: 用户使用云端 API 时网络中断

**修复建议**:
- 添加请求超时 (30s)
- 添加重试机制 (最多 3 次)
- 添加离线模式提示

## 9. 总结

### 核心结论

CodePilot vNext 修订方案在安全性上取得**显著进步**，但仍需在 Week 5-7 完成关键安全加固才能进入实施阶段。

**三大关键改进**:
1. 5 个 P0 漏洞已纳入修复计划
2. 砍掉协作功能，攻击面减少 40%
3. POC 验证包含安全测试

**三大剩余风险**:
1. 模型架构不明确 (P1 阻塞项)
2. 低配设备资源耗尽 (P2)
3. 向量注入防御不足 (P2)

### 给团队的建议

**给 Tech Lead**:
- Week 5 必须明确模型架构，这是安全和技术的共同阻塞项
- POC 验证必须包含安全测试用例，不能仅测性能

**给 Product Director**:
- 砍掉协作功能是正确决策，显著降低安全风险
- v1.1 引入协作前，必须完成安全设计评审

**给 UX Director**:
- 安全限制可能影响性能，需要设计优雅降级 UI
- 低配设备警告需要友好的文案，不要吓跑用户

**给 Domain Expert**:
- 向量检索限制在 Chat/搜索场景是正确的
- 建议补充向量验证的技术规范
