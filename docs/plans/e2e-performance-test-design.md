## 3. 端到端测试用例

### 3.1 基础工作流测试

**测试 1: 单文件变更触发解析**
- 创建测试文件
- 启动服务监听
- 修改文件内容
- 验证：解析被触发，缓存更新

**测试 2: 批处理去重**
- 快速连续修改同一文件 3 次
- 验证：只触发 1 次解析（100ms 窗口内去重）

**测试 3: 多文件批处理**
- 同时修改 5 个文件
- 验证：批量解析，单次批处理事件

### 3.2 增量解析测试

**测试 4: 增量解析复用 AST**
- 首次解析文件
- 修改文件部分内容
- 验证：复用旧 AST，仅解析变更部分

**测试 5: 缓存命中**
- 解析文件 A
- 解析文件 B
- 再次访问文件 A
- 验证：从缓存读取，无重复解析

### 3.3 边界情况测试

**测试 6: 大文件处理**
- 创建 1MB+ 的 TypeScript 文件
- 验证：正常解析，不超时

**测试 7: 缓存淘汰**
- 解析 101 个文件（超过缓存上限 100）
- 验证：最旧的文件被淘汰

**测试 8: 文件删除**
- 解析文件后删除
- 验证：缓存清理，无内存泄漏

## 4. 性能测试用例

### 4.1 吞吐量测试

**测试 9: 批量文件处理**
- 同时修改 50 个文件
- 测量：总处理时间，平均每文件耗时
- 目标：< 5 秒完成 50 个文件

**测试 10: 持续负载**
- 每秒修改 10 个文件，持续 30 秒
- 测量：吞吐量稳定性，无性能退化
- 目标：稳定处理，无积压

### 4.2 延迟测试

**测试 11: 增量解析延迟**
- 修改单个文件
- 测量：从文件变更到解析完成的时间
- 目标：< 100ms（含批处理窗口）

**测试 12: 缓存命中延迟**
- 从缓存读取已解析文件
- 测量：缓存查询时间
- 目标：< 1ms

### 4.3 资源使用测试

**测试 13: 内存使用**
- 解析 100 个文件
- 测量：内存占用峰值
- 目标：< 500MB

**测试 14: CPU 使用**
- 批量解析时监控 CPU
- 测量：CPU 使用率
- 目标：< 80% 单核

## 5. 测试实现策略

### 5.1 测试工具
- **vitest**: 测试框架
- **fs/promises**: 文件操作
- **performance.now()**: 性能计时
- **process.memoryUsage()**: 内存监控

### 5.2 辅助函数

```typescript
// 创建测试文件
async function createTestFile(path: string, size: 'small' | 'large'): Promise<void>

// 等待文件监听触发
async function waitForParse(timeout: number = 500): Promise<void>

// 测量执行时间
async function measureTime<T>(fn: () => Promise<T>): Promise<{ result: T; time: number }>

// 生成测试代码
function generateCode(lines: number): string
```

### 5.3 测试隔离
- 每个测试使用独立的临时目录
- 测试间互不干扰
- 自动清理测试文件
